# Проверка телефонии через OpenVPN, 2026-07-06

## Короткий вывод

Пропустить телефонию через VPN технически можно: OpenVPN-хост `192.168.100.30` уже видит телефонный сегмент и UCM `192.168.0.221` через MikroTik `192.168.100.250`.

Но для OpenVPN-клиентов это еще не готово: сеть `192.168.0.0/24` отсутствует в управляемых сетях OpenVPN и не входит ни в один шаблон. Клиентам сейчас не будет автоматически выдаваться маршрут до телефонии.

## Что уже проверено

- С OpenVPN-хоста маршрут до UCM идет через MikroTik:
  - `192.168.0.221 via 192.168.100.250 dev ens18 src 192.168.100.30`.
- Ping до UCM LAN проходит:
  - `192.168.0.221`: 3/3 ответа, 0% потерь.
- TCP-порты UCM LAN:
  - `192.168.0.221:80` открыт;
  - `192.168.0.221:5060` открыт;
  - `192.168.0.221:5061` открыт;
  - `192.168.0.221:8089` открыт;
  - `192.168.0.221:443` отвечает `ConnectionRefusedError`.
- UCM WAN `192.168.5.2` с OpenVPN-хоста на портах `80/443/5060/5061/8089` не отвечает по TCP.
- В `vpnctl networks list` нет сети `192.168.0.0/24`.
- В `vpnctl network-templates list` нет сети `192.168.0.0/24`.

## Что видно по MikroTik

По read-only проверке RouterOS в этой сессии:

- На MikroTik есть адрес `192.168.0.250/24` на `bridge-lan`, комментарий `Old phone network`.
- Есть активный обратный маршрут к OpenVPN-пулу:
  - `192.168.50.0/24 -> 192.168.100.30`, комментарий `OpenVPN new pool`.
- Телефонная сеть `192.168.0.0/24` для MikroTik является connected-сетью через `bridge-lan`.
- Явных правил firewall для `192.168.50.0/24 <-> 192.168.0.0/24` перед `Drop All` не найдено.
- SIP service helper на MikroTik был включен:
  - `ip firewall service-port sip disabled=false`.

## Что нужно добавить для VPN-доступа к телефонии

1. В OpenVPN Web добавить сеть:
   - CIDR: `192.168.0.0/24`;
   - tag: `telephony`;
   - NAT: `false`;
   - комментарий: `Grandstream UCM / phone LAN`.
2. Создать отдельный шаблон, например `telephony_vpn`, с сетью `192.168.0.0/24`.
3. Для клиентов, которым нужна телефония, назначать этот шаблон или добавить телефонную сеть в нужный служебный шаблон.
4. После применения шаблона выполнить синхронизацию CCD/OpenVPN, чтобы клиент получил маршрут.
5. На MikroTik добавить явные allow-правила выше `Drop All`:
   - `192.168.50.0/24 -> 192.168.0.0/24`;
   - `192.168.0.0/24 -> 192.168.50.0/24`.
6. Для SIP через VPN отключить SIP ALG/helper на MikroTik.
7. В UCM добавить VPN-подсеть в SIP NAT local networks, если SIP-телефоны будут регистрироваться через VPN:
   - минимум OpenVPN-пул `192.168.50.0/24`;
   - отдельно подсеть второго здания, если будет site-to-site.

## Недостающие данные

- RTP-диапазон UCM, чтобы явно разрешить голосовой медиатрафик, а не только SIP `5060/5061`.
- Точная подсеть второго здания, если речь не про одиночных OpenVPN-клиентов, а про site-to-site.
- Нужно ли давать VPN-доступ только к `192.168.0.0/24` или также к WAN UCM `192.168.5.0/24`.
- Подтверждение, что SIP ALG можно отключать на MikroTik без влияния на текущие внешние SIP-транки.
- Физический статус FXO-линий `351-44-9-03-19` и `351-44-9-02-52`.
- Существование реального HT813 или подтверждение, что trunk `HT813_73514490324` устарел.
- Причины незарегистрированных внутренних номеров `109`, `115`, `122`, `124`.
