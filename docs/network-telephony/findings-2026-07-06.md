# Проверка телефонии через OpenVPN, 2026-07-06

## Короткий вывод

Пропустить телефонию через VPN технически можно: OpenVPN-хост `192.168.100.30` уже видит телефонный сегмент и UCM `192.168.0.221` через MikroTik `192.168.100.250`.

Статус после донастройки: сеть `192.168.0.0/24` добавлена в каталог OpenVPN, созданы шаблоны `telephony_vpn` и `lan_full_telephony`. Удаленная подсеть за MikroTik-роутером определена как `192.168.99.0/24`; она добавлена в каталог OpenVPN как `remote-site`, а в `server.conf` добавлен site-route `route 192.168.99.0 255.255.255.0`.

Для полноценного site-to-site нужен конкретный OpenVPN common name клиента роутера: в его CCD должен быть `iroute 192.168.99.0 255.255.255.0`. Обычный сетевой шаблон для этого не подходит, потому что переписывает CCD и не сохраняет `iroute`.

## Что уже проверено

- С OpenVPN-хоста маршрут до UCM идет через MikroTik:
  - `192.168.0.221 via 192.168.100.250 dev ens18 src 192.168.100.30`.
- Ping до UCM LAN проходит:
  - `192.168.0.221`: 3/3 ответа, 0% потерь.
- TCP-порты UCM LAN:
  - `192.168.0.221:80` открыт;
  - `192.168.0.221:5060` открыт;
  - `192.168.0.221:5061` открыт;
  - `192.168.0.221:8089` открыт;
  - `192.168.0.221:443` отвечает `ConnectionRefusedError`.
- UCM WAN `192.168.5.2` с OpenVPN-хоста на портах `80/443/5060/5061/8089` не отвечает по TCP.
- В `vpnctl networks list` есть сеть `192.168.0.0/24` с тегом `telephony`, NAT выключен.
- В `vpnctl network-templates list` есть:
  - `telephony_vpn`: только `192.168.0.0/24`;
  - `lan_full_telephony`: `192.168.100.0/23`, `10.83.1.0/24`, `10.10.10.0/23`, `192.168.0.0/24`, DNS включен.
- В `vpnctl site-routes list` есть:
  - `192.168.99.0/24`: `route 192.168.99.0 255.255.255.0`.

## Что видно по MikroTik

По read-only проверке RouterOS в этой сессии:

- На MikroTik есть адрес `192.168.0.250/24` на `bridge-lan`, комментарий `Old phone network`.
- Есть активный обратный маршрут к OpenVPN-пулу:
  - `192.168.50.0/24 -> 192.168.100.30`, комментарий `OpenVPN new pool`.
- Телефонная сеть `192.168.0.0/24` для MikroTik является connected-сетью через `bridge-lan`.
- Явные firewall allow-правила для `192.168.50.0/24 <-> 192.168.0.0/24` стоят выше `Drop All`.
- SIP service helper на MikroTik отключен:
  - `ip firewall service-port sip disabled=true`.
- Для удаленной подсети `192.168.99.0/24` на центральном MikroTik пока не найдено:
  - маршрута `192.168.99.0/24 -> 192.168.100.30`;
  - явных firewall allow-правил `192.168.0.0/24 <-> 192.168.99.0/24`.

## Что нужно добавить для VPN-доступа к телефонии

1. Выбрать или создать конкретный OpenVPN-клиент роутера для удаленной сети `192.168.99.0/24`.
2. Клиент роутера должен быть типа `router_site_to_site` с `remote_lan_cidr=192.168.99.0/24`, чтобы в CCD появился `iroute`.
3. На UCM добавить SIP NAT local network `192.168.99.0/24`, потому что SIP-телефоны/FXS за удаленным MikroTik будут иметь адреса из этой подсети.
4. На UCM добавить статический маршрут к удаленной подсети:
   - destination: `192.168.99.0`;
   - mask: `255.255.255.0`;
   - gateway: `192.168.0.250`;
   - interface: LAN.
5. На центральном MikroTik добавить маршрут `192.168.99.0/24 -> 192.168.100.30` и allow-правила между `192.168.0.0/24` и `192.168.99.0/24`.
6. На удаленном MikroTik не делать NAT между `192.168.99.0/24` и центральной телефонией; добавить статический маршрут к `192.168.0.0/24` через OpenVPN-интерфейс.
7. После настройки клиента роутера выполнить рестарт OpenVPN в окно работ, чтобы site-route из `server.conf` стал активным.
8. Снять RTP-диапазон UCM и проверить звонок с голосом в обе стороны.

## Недостающие данные

- RTP-диапазон UCM, чтобы явно разрешить голосовой медиатрафик, а не только SIP `5060/5061`.
- Точный OpenVPN common name клиента удаленного MikroTik, за которым будет `192.168.99.0/24`.
- Нужно ли давать VPN-доступ только к `192.168.0.0/24` или также к WAN UCM `192.168.5.0/24`.
- Подтверждение статического маршрута UCM к `192.168.99.0/24` через `192.168.0.250`; по скриншоту вкладка статических маршрутов выглядела пустой.
- Физический статус FXO-линий `351-44-9-03-19` и `351-44-9-02-52`.
- Существование реального HT813 или подтверждение, что trunk `HT813_73514490324` устарел.
- Причины незарегистрированных внутренних номеров `109`, `115`, `122`, `124`.
