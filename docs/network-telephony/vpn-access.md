# VPN-доступ к телефонии

Цель: разрешить OpenVPN-клиентам доступ к телефонному сегменту `192.168.0.0/24` и к UCM `192.168.0.221`, чтобы IP-телефоны или FXS-шлюзы могли работать через VPN.

## Требования

- Двусторонняя маршрутизация между VPN-пулом и телефонным сегментом.
- Без NAT между зданиями для SIP/RTP.
- SIP ALG отключен на промежуточных роутерах.
- Разрешены SIP `5060/5061` и RTP-диапазон UCM.
- Подсеть удаленного здания добавлена в локальные SIP NAT-сети UCM.
- Для аналоговых телефонов во втором здании нужен FXS-шлюз HT814/HT818 через VPN.
- Для внешней медной линии во втором здании нужен FXO-шлюз, например HT813 или GXW4104/GXW4108.

## Что должно быть на OpenVPN

- В профиле клиента должен быть маршрут `192.168.0.0/24`.
- Если нужен доступ к WAN-интерфейсу UCM, отдельно проверить маршрут `192.168.5.0/24`; не добавлять без явной причины.
- CCD/шаблон клиента должен включать телефонную сеть как разрешенную сеть.

## Что должно быть на MikroTik

- Обратный маршрут к OpenVPN-пулу через Ubuntu/OpenVPN-хост `192.168.100.30`.
- Firewall forward должен разрешать:
  - `192.168.50.0/24 -> 192.168.0.0/24`;
  - `192.168.0.0/24 -> 192.168.50.0/24`.
- Для site-to-site VPN второго здания аналогично разрешить подсеть второго здания.

## Что проверять

- С OpenVPN-хоста:
  - `ping 192.168.0.221`;
  - TCP `5060`, `5061`, `80/443` UCM при необходимости;
  - UDP SIP/RTP полноценно проверяется только реальным телефоном или SIP-клиентом.
- С VPN-клиента:
  - есть маршрут на `192.168.0.0/24`;
  - ping/доступ к `192.168.0.221`;
  - регистрация SIP-телефона на UCM;
  - звонок коротким номером, например `101 -> 105`.
